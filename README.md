# Funeral Home Software — Transport Dispatch

Mortuary transport dispatch and chain-of-custody tracking. Two surfaces from one codebase:

- **Dispatcher dashboard** (desktop) — case intake, assignment, live map, audit log, user/vehicle/facility admin.
- **Driver app** (mobile / installable PWA) — run list, navigation, status updates, signatures, photos.

## Stack

- **Frontend**: React 19 + TanStack Start (Vite 7), Tailwind v4, shadcn/ui.
- **Backend**: Lovable Cloud (Supabase Postgres + Auth + Storage + Realtime).
- **Server logic**: TanStack `createServerFn` (not Supabase Edge Functions). Files: `src/lib/*.functions.ts` with `requireSupabaseAuth` middleware where auth is needed; admin work uses `supabaseAdmin` from `client.server.ts`.
- **Hosting**: Cloudflare Workers (edge). Node-incompatible packages are off-limits — see `src/integrations/supabase/` and avoid `child_process`, `sharp`, native addons, etc.

## Auth & Roles

- Email/password via Supabase Auth. New signups land on `/pending-approval` until an admin flips `profiles.approved`.
- Roles live in `public.user_roles` (enum `app_role`: `admin | dispatcher | driver | viewer`). **Never** stored on `profiles`.
- Access is checked via SECURITY DEFINER functions `has_role(uuid, app_role)` / `has_any_role(uuid, app_role[])` / `is_case_driver(uuid, uuid)`. The linter flags these as "search_path" warnings — accepted (see `mem://security/role-check-functions`).
- Route gates:
  - `src/routes/_authenticated.tsx` — requires session + `approved=true`.
  - `src/routes/_authenticated/_dispatcher.tsx` — admin/dispatcher/viewer.
  - `src/routes/_authenticated/driver.tsx` — driver (or admin acting as driver).

## Data Model (high level)

Public schema tables: `profiles`, `user_roles`, `cases`, `case_events`, `case_signatures`, `case_documents`, `facilities`, `vehicles`, `drivers` (assignment), `driver_locations`, `push_subscriptions`, `audit_log`.

- **RLS is on for every table.** Drivers only see cases they're assigned to (`is_case_driver`). Dispatchers/admins see all. Viewers are read-only.
- **Realtime** is enabled on: `cases`, `case_events`, `driver_locations`, `profiles`.
- **Storage**: private bucket `case-documents`. All access via short-lived signed URLs — never public URLs.

## Sensitive Data

Decedent information is sensitive. Apply minimum-necessary access in any new query, policy, or server function. Formal HIPAA/BAA work is planned for v2 — flag anything that would block that.

## Repository Scope

This repo contains **only** the funeral home dispatch app. It does not include any unrelated Lovable projects (e.g. closet/wardrobe app) — those live in separate Lovable projects and separate GitHub repos.

## Local Dev

```bash
bun install
bun run dev
```

`.env` is auto-managed by Lovable Cloud (`VITE_SUPABASE_URL`, `VITE_SUPABASE_PUBLISHABLE_KEY`, `VITE_SUPABASE_PROJECT_ID`). Server-only secrets (`SUPABASE_SERVICE_ROLE_KEY`, etc.) are injected at runtime — do not commit.

## Review Focus for Codex

Suggested areas for engineering review:

1. **RLS correctness** — verify every table's policies match intended role access; check for policy gaps on insert/update vs select.
2. **Server function auth** — confirm every `createServerFn` that touches user data uses `requireSupabaseAuth` (or has an explicit reason not to).
3. **Signed URL handling** — `case-documents` bucket; ensure no code path returns public URLs or long-lived signed URLs.
4. **Realtime channels** — confirm subscriptions are scoped + cleaned up, and that RLS still gates the payload.
5. **PII / decedent data** — minimum-necessary in selects; PII purge on signout already clears `case-intake-draft:*` from sessionStorage.
6. **Edge runtime compatibility** — flag any Node-only deps creeping in.
7. **Error handling & retries** — server functions, push notifications, location updates.
8. **Scalability** — indexes on hot paths (`cases.status`, `cases.assigned_driver_id`, `driver_locations.driver_id`, `case_events.case_id`).
